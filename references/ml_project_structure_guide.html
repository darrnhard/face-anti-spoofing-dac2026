Section 01
Directory Structure Cookiecutter DS Standard

The gold standard is Cookiecutter Data Science (CCDS) by DrivenData — adopted by Kaggle grandmasters, academic research groups, and production ML teams. The core principle: separate exploration code (notebooks) from reusable code (src/), and treat data as immutable at source.

fas-competition/ ← Professional CCDS-based layout

│

⚡ Key principle (CCDS): data/raw/ is the immutable truth. Never overwrite it. All transformations write to data/interim/ or data/processed/. Scripts in src/ are the source of truth — not notebooks. Notebooks are for exploration, scripts are for reproducibility.
💡 For solo Kaggle use, you can drop docs/, references/, and Makefile initially. The non-negotiables are: data/raw, src/, configs/, notebooks/experiments/, and submissions/.
Section 02
File Naming Conventions Industry Standard

Consistent naming is not aesthetic preference — it's what lets you parse 50 files instantly without opening them. Different file types have different conventions.

📓 Notebooks
PatternExampleNotes
[stage]-[desc].ipynb 01-eda-class-balance.ipynb Cookiecutter DS standard. Stage # controls ordering. Dash-delimited, lowercase.
[expNN]-[model]-[tweak].ipynb exp03-effnet-b4-mixup.ipynb For experiment notebooks. Zero-padded number ensures sorting. Use exp prefix to immediately distinguish from EDA.
scratch-[desc].ipynb scratch-debug-augment.ipynb Throwaway code. scratch- prefix = "don't clean this up"
[anything]-v2.ipynb notebook_v2_final_FINAL.ipynb Anti-pattern. Version suffixes on notebooks are a red flag — use Git commits or experiment numbering instead.
🐍 Python Source Files (src/)
PatternExampleNotes
snake_case.py face_dataset.py Always snake_case for Python files. PEP 8 standard.
[function]_[noun].py augmentation_factory.py Name files after what they contain: a factory, a dataset, a trainer.
train.py / predict.py train.py, evaluate.py, predict.py CLI scripts use action verbs. Run as python scripts/train.py.
⚙️ Config Files (configs/)
PatternExampleNotes
base.yaml base.yaml Default hyperparameters. All experiments override from here.
[expNN]_[model]_[tweak].yaml exp03_effnet_b4_mixup.yaml Matches the experiment notebook name. YAML only stores the delta from base.
💾 Model Checkpoints
PatternExampleNotes
best_[metric][score].ckpt best_val_auc0.951.ckpt Best checkpoint uses metric + score. Instantly scannable without opening.
epoch=[N]-step=[N].ckpt epoch=12-step=2340.ckpt PyTorch Lightning default — good for resume training checkpoints.
[expNN]/best_*.ckpt exp03_effnet_b4_mixup/best_val_auc0.951.ckpt Always nest checkpoints inside an experiment-named folder. ALWAYS save a config_snapshot.yaml alongside.
📤 Submissions
PatternExampleNotes
sub_[exp]_cv[score]_lb[score].csv sub_exp03_cv0.951_lb0.904.csv The best naming scheme. Shows exactly which experiment, CV score, and LB score at a glance.
sub_[date]_[exp].csv sub_20250402_exp03.csv Alternative if you make multiple submissions per day.
submission.csv / sub_final.csv submission_v3_BEST.csv Anti-pattern. You'll have 20 files and won't know which is which in 2 weeks.
✅ Master rule: A filename should answer "what is this and how good was it?" without opening the file. sub_exp03_cv0.951_lb0.904.csv answers that completely. submission_final2.csv answers nothing.
Section 03
Notebook Splitting Strategy Pipeline Separation

Professional Kaggle competitors (including Silver/Gold medalists) never run everything in one notebook. Each pipeline stage becomes its own notebook. This way: if preprocessing takes 7 hours, you only run it once — and the notebook becomes a cached artifact.

NB 00
EDA
01-eda-class-dist.ipynb
02-eda-sample-viz.ipynb
›
NB 01
Preprocessing
03-preprocess-crops.ipynb
→ saves to data/processed/
›
NB 02
Training
exp01-autogluon.ipynb
exp02-effnet-b0.ipynb
exp03-effnet-b4.ipynb
›
NB 03
Evaluation
04-eval-compare.ipynb
05-error-analysis.ipynb
›
NB 04
Inference
06-inference-submit.ipynb
→ saves to submissions/
🔬 EDA Notebooks
Run once, rarely modified. Keep analysis self-contained. Prefix: 01-eda-. Output is insight, not data. Use markdown cells heavily — this is documentation.
⚙️ Preprocessing Notebooks
The "expensive" stage. Runs once, caches output to data/processed/. Almost never re-run. If your preprocessing takes >30min, this split is non-negotiable. Prefix: 02-preprocess-
🧪 Experiment Notebooks
One notebook per experiment. Each loads from data/processed/, trains, logs metrics, saves checkpoint. Prefix: exp01-, exp02-. Sequential numbering = chronological history.
📊 Eval / Analysis Notebooks
Load multiple trained models, compare metrics, plot confusion matrix, identify failure modes. Prefix: 04-eval-. Helps decide the next experiment direction.
📤 Inference Notebooks
Load the best checkpoint, run on test set, generate submission. Prefix: 05-infer-. Keep this notebook clean and simple — it's your submission machine. Separating it prevents accidental re-training.
🗑️ Scratch Notebooks
Throwaway prototyping only. Keep in notebooks/scratch/. Never number these — they're not part of the pipeline. Prefix: scratch-. Delete freely.
💡 Kaggle grandmaster tip (from TDS article by Ibrahim Habib, Silver Medal NeurIPS 2025): "Separating each step in the pipeline (e.g. data preprocessing, training, inference) into its own notebook is useful when one step takes a long time to run and rarely changes. My preprocessing step took more than seven hours to run."
⚠️ The module import trick: Add this to the top of every notebook so you can import from src/:

import sys, os; sys.path.insert(0, os.path.abspath('../'))
from src.data.dataset import FASDataset

This means you write your class once in src/ and reuse it in all notebooks.
Section 04
Iteration Versioning Git + Experiment Tracking

Professional teams use a two-level versioning system: Git for code versions, and experiment tracking (MLflow / W&B / CSV log) for run/result versions. These serve different purposes and must not be conflated.

LEVEL 1
Code
Git Commits — The Source of Truth
Every meaningful change gets committed. Commit messages follow a pattern. When a Kaggle submission is made, tag the commit with the LB score. This lets you time-travel to any previous model exactly.
git commit -m "exp03: add MixUp augmentation, EfficientNet-B4"
git commit -m "fix: correct label leak in validation split"
git tag "lb-0.904-exp03" HEAD
LEVEL 2
Experiments
Experiment IDs — Sequential, Never Overwritten
Each training run gets a sequential ID: exp01, exp02, etc. The experiment log records what changed. Never delete or overwrite an experiment — even bad ones are informative. The log is a lab notebook.
# experiment_log.md | ID | Model | Augment | CV ACER | LB ACER | Notes | |-------|---------------|---------------|----------|----------|------------------------| | exp01 | AutoGluon MM | default | 0.142 | 0.189 | Baseline AutoML | | exp02 | EfficientB0 | flip+crop | 0.091 | 0.109 | Custom PyTorch | | exp03 | EfficientB4 | +MixUp | 0.049 | 0.096 | Best so far ✓ |
LEVEL 3
Configs
YAML Configs — Hyperparameter Snapshots
Every experiment has a YAML config that fully specifies it. The config is version-controlled with Git. The checkpoint folder always contains a frozen copy of the config at time of training. This is what makes results reproducible 6 months later.
# configs/experiments/exp03_effnet_b4_mixup.yaml _base_: ../base.yaml model: name: efficientnet_b4 pretrained: true training: epochs: 30 lr: 1e-4 batch_size: 32 augmentation: mixup_alpha: 0.4 use_cutmix: true
LEVEL 4
Checkpoints
Model Checkpoints — Named by Metric Score
Checkpoints are saved in an experiment-named folder. The best checkpoint includes the validation metric in its name. A config snapshot lives beside it. This way, if you have 10 experiments, you immediately know which folder holds your best model.
models/ ├── exp02_effnet_b0/ │ ├── best_val_auc0.934.ckpt │ └── config_snapshot.yaml └── exp03_effnet_b4_mixup/ ├── best_val_auc0.951.ckpt ← Best model at a glance └── config_snapshot.yaml ← Reproducibility key
🚫 Anti-patterns to eliminate from your codebase now: model_v2.ckpt, model_final.ckpt, model_new.ckpt, train_copy.py, notebook_FINAL_v3.ipynb. These will haunt you. Use experiment IDs + metric scores instead — they are self-describing.
Section 05
Experiment Tracking Tools Tooling

You need to track: hyperparameters, metrics per epoch, code version (git hash), artifacts. Tool choice depends on your complexity level.

W&B (wandb)
The industry favorite for DL research. Logs metrics, gradients, images, model topology. Beautiful dashboards. Free for personal use. Integrates with PyTorch, Lightning, Keras in ~3 lines. Best choice for your FAS project.
RECOMMENDED
MLflow
Open-source, self-hosted, no account needed. Tracks params, metrics, artifacts. Slightly more setup. Good for offline/private work or when you don't want cloud dependency.
OPTIONAL
experiment_log.md
A simple markdown table in your repo. Manual, but zero-friction. Even if you use W&B, keep a brief manual log for high-level notes ("this failed because of X"). The human-readable audit trail.
ALWAYS DO
Hydra (configs)
Facebook Research config management. Composes YAML configs, sweeps hyperparameters from CLI, logs configs per run automatically. Used by the PyTorch Lightning community template (lightning-hydra-template on GitHub).
OPTIONAL
DVC
Git for large files (data + model checkpoints). Versions your datasets alongside your code. Overkill for solo Kaggle, essential for team or production ML. Use if your data changes across experiments.
LATER
✅ Minimum viable tracking setup for FAS competition: W&B for live metric logging + experiment_log.md for high-level decisions + Git tags on submission commits. That's 90% of the value with 10% of the complexity.
Section 06
The Scripts vs Notebooks Philosophy Core Principle

The most important professional principle in ML engineering. Professionals treat notebooks and scripts as fundamentally different tools with non-overlapping roles.

📓 Notebooks are for...
  • Exploratory Data Analysis (EDA)
  • Quickly prototyping a new idea
  • Visualizing results and errors
  • Creating figures for reports
  • Running a single experiment interactively
  • Communication (markdown-heavy)
🐍 Scripts (src/) are for...
  • Reusable Dataset / DataModule classes
  • Model architecture definitions
  • Training/evaluation loops
  • Loss functions and metrics
  • Any code used in 2+ notebooks
  • Reproducible pipeline execution
🔄 The refactoring workflow: Start in a notebook → prototype works → extract the reusable function/class to src/ → import it back into the notebook. This is the professional loop. The notebook becomes thin; src/ becomes the real codebase. When a bug is found in src/dataset.py, it's fixed once and all notebooks that import it are immediately fixed too.
🚫 The classic mistake: Copying the same preprocessing block across 5 different experiment notebooks. When a bug is found, you fix it 5 times (or miss some). This is exactly what happens when the codebase grows without structure. The fix is extracting to src/data/preprocessing.py.
